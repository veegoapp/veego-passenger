## Phase 2 — Notification & Privacy Preferences

### Database schema additions (users table)

Seven new boolean columns, all default `true` (opted in):

```
notif_trips           boolean  default true  — trip/booking/shuttle notifications
notif_promos          boolean  default true  — promo/offer notifications

notif_driver_updates  boolean  default true  — driver-initiated passenger notifications
```

## Phase 3 — Avatar Upload & Notification Gating

### New endpoint

| Method | Path | Auth | What it does |
|--------|------|------|-------------|
| `POST` | `/users/me/avatar` | passenger | Upload avatar image to Supabase; returns permanent CDN URL |

**Request:** `multipart/form-data`, field name `avatar`, accepted: JPEG/PNG/WebP, max 5 MB  
**Response `200`:** `{ "avatarUrl": "https://..." }`  
**Notes:**
- The CDN URL is permanent — no refresh needed
- After upload, `GET /users/me` returns the updated `avatar` field
- Supabase bucket must exist and be set to **public** (bucket name: `avatars`)

### Notification gating

All `notification:new` socket emits to passenger rooms are now gated by the user's preference flags set in Phase 2. **19 passenger emit sites gated across 9 files.**

Summary of which flag gates which events:

| Flag | What it gates |
|------|--------------|
| `notifTrips` | Trip/shuttle status, booking confirmations, booking cancellations, driver no-show refunds, post-trip rating prompts |
| `notifDriverUpdates` | Absence warnings and fines applied by the driver |
| `notifSystem` | Wallet refunds (admin-initiated), admin broadcast messages |
| `notifPromos` | Reserved — no promo notifications in codebase yet |

Gating is **fail-open**: if the DB preference check fails transiently, the notification is still delivered.

## Key Socket Events for Passenger App

### Ride lifecycle

```
ride:driver_assigned    — { rideId, driverId, driverName, driver: { name, phone, vehicle, rating }, eta }
ride:driver_location    — { rideId, location: { latitude, longitude }, timestamp }
ride:arrived            — { rideId, driverId }            ← use this (was ride:driver_arrived)
ride:started            — { rideId, driverId }
ride:completed          — { rideId, finalPrice, fare, waitingCharge }
ride:cancelled          — { rideId, cancelledBy, reason, cancellationFee }
ride:driver_cancelled   — { rideId, message }
ride:no_show_cancelled  — { rideId, reason, refundAmount, cancellationFee }  ← new (was driver-only)
ride:timeout            — { rideId, reason: "timeout", message }              ← use this (was ride:status_update)
```

### Notifications

```
notification:new  — { id, category, title, body, time, ...extra }
```

Received only when the user's relevant preference flag is `true`.

### Shuttle rooms

```
join:trip    — join the passenger to trip room
leave:trip   — remove from trip room
shuttle:trip:status — { tripId, status, stationProgress }
```

## Infrastructure Notes

- **Node 20 + Supabase**: Supabase realtime requires a WebSocket constructor. Fixed by passing the `ws` package as the transport. This is already resolved in the codebase — no action needed.
- **Notification gating DB cost**: Each gated notification emit makes one extra DB query (SELECT 4 columns from `users` by id). These are single-row indexed lookups and add negligible latency. The admin broadcast batches all preference checks in a single query.
- **Privacy flags**: Backend honours these flags for GPS trace logging and analytics suppression. The flags are forward-looking; no historical data is removed when a user opts out.
