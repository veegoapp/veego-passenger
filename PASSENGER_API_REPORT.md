## 1. Authentication

### `POST /auth/login`
| Field | Value |
|---|---|
| **File** | `app/auth.tsx` |
| **Auth** | None |
| **Purpose** | Email + password login |

**Request body:**
```json
{ "email": "string", "password": "string" }
```

**Response fields accessed:**
- `data.accessToken / data.access_token / data.token`
- `data.refreshToken / data.refresh_token`
- `data.user.name`
- `data.requiresOtp` → if true, redirect to `/verify-phone` with `phone` + `maskedPhone`
- `data.error`, `data.message`

**Error codes:**
- `403` with `requiresOtp: true` → OTP verification required
- `403` with code `account_blocked` → account suspended

---

### `POST /auth/register`
| Field | Value |
|---|---|
| **File** | `app/auth.tsx` |
| **Auth** | None |
| **Purpose** | New user registration |

**Request body:**
```json
{ "name": "string", "phone": "string", "email": "string", "password": "string" }
```

**Response fields accessed:**
- `data.requiresOtp` → if true, go to OTP step (passes `termsVersion` param)
- `data.phone`, `data.maskedPhone`
- Tokens: `data.accessToken / data.access_token / data.token`, `data.refreshToken`

---

### `POST /auth/send-otp`
| Field | Value |
|---|---|
| **File** | `app/verify-phone.tsx` |
| **Auth** | None |
| **Purpose** | Send / resend OTP to phone |

**Request body:**
```json
{ "phone": "string" }
```

**Error codes:**
- `429` with `data.retryAfter` (number of seconds) → rate limited

---

### `POST /auth/verify-otp`
| Field | Value |
|---|---|
| **Files** | `app/verify-phone.tsx`, `app/auth.tsx` (forgot-password flow) |
| **Auth** | None |
| **Purpose** | Verify 6-digit OTP (used in both registration and forgot-password flows) |

**Request body:**
```json
{ "phone": "string", "otp": "string" }
```

**Response fields accessed (forgot-password):**
- `data?.token ?? data?.resetToken ?? data?.data?.token`

**Response fields accessed (registration):**
- `data.accessToken / data.access_token / data.token`
- `data.refreshToken / data.refresh_token`
- `data.user.email / data.user.phone / data.user.name`

---

### `POST /auth/forgot-password`
| Field | Value |
|---|---|
| **File** | `app/auth.tsx` |

| **Purpose** | Initiate password reset (sends OTP to phone) |

**Request body:**
```json

```

### `POST /auth/reset-password`
| Field | Value |


**Notes:** Token comes from `POST /auth/verify-otp` response in forgot-password flow.

## 3. Bookings

### `POST /bookings`
| Field | Value |
|---|---|
| **Files** | `src/api/shuttleService.ts`, `context/BookingContext.tsx` |
| **Auth** | JWT |
| **Purpose** | Create a new shuttle seat booking |

**Request body:**
```json
{
  "tripId": "number",
  "seatCount": "number",
  "paymentMethod": "cash",
  "promoCode": "string (optional)"
}
```

**Notes (shuttleService.ts):** `seatCount` is always hardcoded to `1`.  
**Notes (BookingContext.tsx):** `seatCount` uses the `seatCount` state (default 1, user-selectable).

**Response fields accessed (BookingContext):**
- `data.id ?? data.booking.id`
- `data.status`
- `data.shuttle` (ShuttleBookingMeta object)

**Error codes:**
- `409` → duplicate booking or race condition (seat taken)

## 18. Analysis

### Duplicate APIs

| Endpoint | Called From | Notes |
|---|---|---|

| `GET /shuttle/lines/:id` | `shuttleService.ts` + `BookingContext.tsx` (`openRoute`) + `BookingContext.tsx` (`fetchLineTrips`) | Three calls, two in the same context |

| `GET /users/me` | `useProfile.ts` + `useNotifications.ts` | Called in two hooks to get user ID and profile separately |

| `GET /services/control` | `ServiceControlContext.tsx` (lines 137 + 236) | Same context calls it on mount and again on manual refresh |

---

### Inconsistent API Usage

| Issue | Details |
|---|---|
| **Two booking cancellation endpoints** | `PATCH /bookings/:id/cancel` (trips.tsx) and `DELETE /shuttle/bookings/:id` (shuttleService.ts) both cancel bookings. Response shapes are different (`{ reason? }` body vs no body; `{ ok, refunded }` vs unhandled). |
| **Driver-namespaced endpoint in passenger app** | `GET /driver/trips/:id/stations` is called from `app/trip-detail.tsx` — a passenger screen. Should be a passenger-scoped endpoint. |
| **Response field fallback chains** | Multiple endpoints use 3–5 fallback keys (e.g., `data.balance ?? data.walletBalance ?? data.wallet_balance ?? data.amount`). Indicates the API response shape was changed without updating the app, or was never formally documented. |
| **`paymentMethod` hardcoded to `'cash'`** | Both `POST /rides/request` and `POST /bookings` hardcode `paymentMethod: 'cash'`. Wallet payment is implemented in the wallet module but never wired to the payment body. |
| **`seatCount` inconsistency** | `shuttleService.ts` hardcodes `seatCount: 1`; `BookingContext.tsx` passes the user-selected value. The service layer ignores the state. |
| **`/user/` vs `/users/me/`** | Profile endpoints use `/users/me/*` (plural) while ratings and locations use `/user/*` (singular). No consistency. |

---

### Potentially Broken or Incomplete APIs

| Endpoint | Risk |
|---|---|
| `GET /driver/trips/:id/stations` | Called from passenger app using a driver-role endpoint. May fail if the server validates role or if a passenger's JWT doesn't have driver permissions. |
| `POST /rides/request` — `paymentMethod: 'cash'` hardcoded | If the server enforces wallet balance deduction when `paymentMethod: 'wallet'` is expected, this will always bypass it. |
| `POST /api/terms/accept` — called before token is stored in some paths | In the OTP registration flow, `acceptTerms()` is called from `verify-phone.tsx` immediately after `persistTokens()`. If the interceptor hasn't picked up the new token yet, the POST may go out unauthenticated. |
| `GET /config/payment-methods` | Response fields are never accessed in the extracted code — the data is fetched but not read. |
| `booking:boarded` socket payload | Server sends `{ bookingId: number, tripId: number, timestamp: string }`. `ticket.tsx` was comparing `bookingId` as string with strict equality — fixed, now uses `String()`. `trip-detail.tsx` ignores payload entirely (no ID check). |

### Key Issues Found

1. **Dead service layer** — `shuttleService.ts` defines `getTrip`, `getRouteStations`, `getMyDebt`, and `createBooking` but none are called by the app. The app bypasses the service layer and calls the API client directly.

2. **Driver endpoint in passenger app** — `GET /driver/trips/:id/stations` is called from the passenger-facing trip detail screen. This is a role boundary violation that may work in dev but fail under strict server-side role enforcement.

3. **Two booking cancellation paths** — `PATCH /bookings/:id/cancel` and `DELETE /shuttle/bookings/:id` both cancel bookings from different screens. Only the DELETE version returns a `refunded` status. The two should be consolidated.

4. **Heavy response fallback chaining** — Endpoints like `/wallet`, `/promo/validate`, `/users/me`, and `/shuttle/lines/:id` use 3–5 fallback keys on response fields. This points to undocumented or unstable API contracts.

5. **`paymentMethod` always `'cash'`** — Wallet integration exists in the UI (`/wallet/topup`, balance display) but is never used as a payment method in booking or ride requests.

6. **`GET /users/me/bookings` called 4 times** with inconsistent pagination params — no single source of truth for booking history.
